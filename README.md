# نظام إدارة الزكاة والتبرعات

تطبيق ويب عربي متكامل **RTL** لإدارة دورة الزكاة والتبرعات النقدية والعينية، مبني بواجهة HTML/CSS/JavaScript خفيفة قابلة للنشر مباشرة على **Vercel**، مع قاعدة بيانات ومصادقة وصلاحيات عبر **Supabase**.

## ما الذي يشمله المشروع؟

- تسجيل الدخول وتحميل صلاحيات المستخدم.
- لوحة تحكم بمؤشرات، تنبيهات، رسوم، وآخر العمليات.
- المستخدمون والأدوار والصلاحيات.
- الموزعون والمتبرعون.
- فئات المستفيدين والحالات الصحية.
- دليل المستفيدين مع كشف التكرار وحالات الاعتماد.
- الحملات النقدية والعينية والمختلطة.
- سندات القبض والصرف النقدي.
- الأصناف، أرصدة المخزون، التشغيلات، والصلاحية.
- سندات القبض والصرف العيني.
- السلال الغذائية ومكوناتها.
- إقفال الحسابات وإعادة الفتح وفق الصلاحية.
- تقارير مالية ومخزنية وتشغيلية.
- سجل تدقيق للعمليات الحساسة.
- شاشة اتصال ومزامنة، وصف محلي للمسودات عند ضعف الشبكة.
- إعدادات عامة ونسخة احتياطية لبيانات العرض التجريبي.
- تصميم متجاوب للهاتف والكمبيوتر، خطا Cairo وTajawal، وأيقونات Font Awesome.

## هيكل المشروع

```text
index.html                  الواجهة الرئيسية
css/styles.css              نظام التصميم والاستجابة
js/config.js                إعداد Supabase ووضع العرض
js/app.js                   التنقل والشاشات والتفاعلات
js/data-service.js          طبقة البيانات وقواعد العمل
js/screen-config.js         تعريف الحقول والجداول لكل شاشة
js/demo-data.js             بيانات العرض المحلي
js/offline.js               قائمة المزامنة المحلية
service-worker.js           دعم PWA والتخزين المؤقت
manifest.webmanifest        بيانات تثبيت التطبيق
vercel.json                 إعداد النشر على Vercel
supabase/schema.sql         قاعدة البيانات، RLS، العروض والدوال
supabase/seed.sql           بيانات مرجعية أولية
supabase/functions/         وظائف إنشاء مستخدم وإعادة كلمة المرور
```

## تشغيل العرض التجريبي محلياً

لا يحتاج العرض التجريبي إلى قاعدة بيانات.

```bash
cd zakat_management_system
python3 -m http.server 8080
```

ثم افتح `http://localhost:8080`.

في وضع العرض يمكنك إدخال أي بريد وكلمة مرور من 6 أحرف أو أكثر. البيانات تحفظ في `localStorage` داخل المتصفح.

## ربط Supabase

### 1. إنشاء المشروع وقاعدة البيانات

1. أنشئ مشروعاً جديداً في Supabase.
2. افتح **SQL Editor**.
3. نفّذ `supabase/schema.sql` كاملاً.
4. نفّذ `supabase/seed.sql`.

يفضل تنفيذ الملفات على مشروع جديد لأن `schema.sql` ينشئ أنواع PostgreSQL مخصصة وجداول ودوال من البداية.

### 2. إنشاء أول مدير

من لوحة Supabase افتح **Authentication → Users** وأنشئ مستخدماً بالبريد وكلمة المرور. بعد ذلك نفّذ:

```sql
update public.profiles
set role = 'admin',
    is_active = true,
    full_name = 'مدير النظام',
    username = 'admin'
where email = 'YOUR_ADMIN_EMAIL';
```

### 3. نشر وظائف Edge

بعد تثبيت Supabase CLI وربط المشروع:

```bash
supabase functions deploy create-user
supabase functions deploy reset-user-password
```

الوظيفتان تتحققان من أن المستدعي مدير، وتستخدمان Service Role داخل الخادم فقط. لا تضع Service Role Key في ملفات الواجهة أبداً.

### 4. تعديل إعداد الواجهة

افتح `js/config.js` وعدّل:

```js
window.ZAKAT_CONFIG = {
  supabaseUrl: "https://YOUR_PROJECT.supabase.co",
  supabaseAnonKey: "YOUR_PUBLISHABLE_OR_ANON_KEY",
  demoMode: false,
  // بقية الإعدادات كما هي
};
```

المفتاح الموجود في المتصفح يجب أن يكون **Publishable/Anon** فقط، بينما الحماية الفعلية تعتمد على سياسات **Row Level Security** الموجودة في `schema.sql`.

### 5. إعداد روابط المصادقة

في Supabase Authentication أضف رابط موقع Vercel إلى **Site URL** و **Redirect URLs**، مثل:

```text
https://your-project.vercel.app
https://your-project.vercel.app/**
```

## النشر على Vercel

1. ارفع المجلد إلى مستودع GitHub أو استورده مباشرة إلى Vercel.
2. اختر Framework Preset: **Other**.
3. لا يوجد Build Command.
4. اجعل Output Directory هو `.` أو اتركه فارغاً.
5. انشر المشروع.

ملف `vercel.json` يتولى مسارات التطبيق وإضافة بعض رؤوس الأمان.

## قواعد العمل المنفذة

- المسودات لا تؤثر على الأرصدة.
- الترحيل وحده يثبت الأثر المالي أو المخزني.
- لا يمكن صرف مبلغ أكبر من رصيد الحملة أو الموزع أو سند القبض.
- لا يظهر المستفيد غير المعتمد في قوائم الصرف النهائية.
- يمنع الصرف المتكرر للمستفيد في الحملة نفسها، إلا بسياسة استثناء موثقة لاحقاً.
- لا يمكن ترحيل سلة ناقصة المخزون.
- الصرف العيني يخصم من التشغيلات وفق الأقرب انتهاءً أولاً.
- إلغاء سند مرحل يعكس أثره مع حفظ السبب وسجل التدقيق.
- لا يمكن الإقفال الكامل إذا وجدت عمليات معلقة أو رصيد غير معالج.
- إعادة المزامنة تستخدم مفاتيح idempotency لمنع إنشاء العملية مرتين.
- لا توجد سياسات حذف لسندات مالية مرحلة؛ الإلغاء هو المسار المعتمد.

## افتراضات النسخة الحالية

- اعتماد الصرف مطلوب افتراضياً.
- العمل دون اتصال يسمح بحفظ المسودات فقط، وليس الترحيل النهائي.
- تتبع المخزون يتم بالتشغيلة وتاريخ الصلاحية.
- العملة الافتراضية هي الريال اليمني `YER`.
- صلاحيات العرض والتعديل تطبق في قاعدة البيانات بواسطة RLS، وليس بإخفاء الأزرار فقط.

## فحص ما قبل التشغيل الفعلي

- راجع صلاحيات كل دور مع الإدارة.
- حدد سياسة الصرف المتكرر والاستثناءات.
- حدد طريقة ترحيل الأرصدة واعتماد السندات.
- اختبر النسخ الاحتياطي والاستعادة في Supabase.
- فعّل MFA لحسابات الإدارة عند الحاجة.
- اربط نطاقاً رسمياً وراجع سياسات الخصوصية والاحتفاظ بالبيانات.
- نفّذ اختبارات قبول على سيناريوهات التكرار، تجاوز الرصيد، المخزون الناقص، الإقفال، والمزامنة.

## ملاحظة تقنية

الواجهة تعمل كـ SPA خفيفة بلا مرحلة بناء. تحميل Google Fonts وFont Awesome وChart.js يتم من CDN؛ للتشغيل في بيئة داخلية من دون إنترنت، نزّل هذه الأصول إلى المشروع وعدّل الروابط محلياً.

## المرفقات

ينشئ `schema.sql` حاوية خاصة باسم `zakat-attachments` بحجم أقصى 5 ميجابايت للملف، وتقبل صور JPEG/PNG/WebP وملفات PDF. تحفظ ملفات سندات القبض وإثباتات الصرف في مسار يبدأ بمعرف المستخدم، وتطبق عليها سياسات Storage تمنع الوصول العام.

## اختبارات النسخة المرفقة

تم فحص بناء JavaScript نحوياً، وتجربة تسجيل الدخول التجريبي، وفتح جميع مسارات الواجهة الثمانية عشر بعد شاشة الدخول، وتجربة النوافذ المنبثقة والبحث، وفحص الاستجابة على عرض هاتف 390 بكسل من دون تمرير أفقي. كما تم اختبار قواعد العرض التجريبي لمنع تكرار الهوية، ومنع الصرف لمستفيد غير معتمد، ومنع تجاوز الرصيد، ومنع تكرار الصرف داخل الحملة.
